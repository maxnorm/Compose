## Overview

LibRoyalty provides a robust and composable implementation of ERC-2981 royalty logic, enabling dynamic royalty settings for both specific tokens and default fallbacks. It uses the diamond storage pattern for seamless integration in diamond contracts, ensuring minimal coupling and easy extensibility.


[View Source](https://github.com/maxnorm/Compose/blob/38545161423dd99e31b48b0eff41d260712fe731/src/token/Royalty/LibRoyalty.sol)


<Callout type="info" title="Library Usage">
This library provides internal functions for use in your custom facets. Import it to access shared storage.
</Callout>

## Errors


### ERC2981InvalidDefaultRoyalty


Thrown when default royalty fee exceeds 100% (10000 basis points).



```solidity
error ERC2981InvalidDefaultRoyalty(uint256 _numerator, uint256 _denominator);
```



### ERC2981InvalidDefaultRoyaltyReceiver


Thrown when default royalty receiver is the zero address.



```solidity
error ERC2981InvalidDefaultRoyaltyReceiver(address _receiver);
```



### ERC2981InvalidTokenRoyalty


Thrown when token-specific royalty fee exceeds 100% (10000 basis points).



```solidity
error ERC2981InvalidTokenRoyalty(uint256 _tokenId, uint256 _numerator, uint256 _denominator);
```



### ERC2981InvalidTokenRoyaltyReceiver


Thrown when token-specific royalty receiver is the zero address.



```solidity
error ERC2981InvalidTokenRoyaltyReceiver(uint256 _tokenId, address _receiver);
```

## Usage Example

```solidity
pragma solidity ^0.8.0;

import {LibRoyalty} from &quot;path-to-library/LibRoyalty.sol&quot;;

contract RoyaltyFacet {
    function setDefaultRoyalty(address _receiver, uint96 _feeNumerator) external {
        require(_receiver != address(0), &quot;Invalid receiver address&quot;);
        LibRoyalty.setDefaultRoyalty(_receiver, _feeNumerator);
    }

    function getRoyaltyInfo(uint256 tokenId, uint256 salePrice) external view returns (address, uint256) {
        return LibRoyalty.royaltyInfo(tokenId, salePrice);
    }

    function clearTokenRoyalty(uint256 tokenId) external {
        LibRoyalty.resetTokenRoyalty(tokenId);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Always validate input parameters, especially receiver addresses and royalty fees, to maintain correct and secure state updates.
- Use LibRoyalty&#39;s resetTokenRoyalty to clear token-specific settings only when expecting fallback to default logic.
- Ensure the functions interacting with LibRoyalty are properly secured if administrative actions are included.
</Callout>



## Integration Notes

<Callout type="success" title="Shared Storage">
LibRoyalty uses the diamond storage pattern, ensuring that state changes made through it are immediately visible to other facets querying royalty information. This design enables modular and consistent handling of royalties across a diamond&#39;s facets.
</Callout>
