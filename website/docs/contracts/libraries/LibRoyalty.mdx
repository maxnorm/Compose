## Overview

LibRoyalty provides a modular and efficient way to manage ERC-2981 compliant royalty logic in diamond contracts. It enables setting, querying, and removing both global and token-specific royalty details, ensuring flexibility and compliance with marketplace standards.


[View Source](https://github.com/maxnorm/Compose/blob/7206f045d711880c211b967e80d72745b52f5cfc/src/token/Royalty/LibRoyalty.sol)


<Callout type="info" title="Library Usage">
This library provides internal functions for use in your custom facets. Import it to access shared storage.
</Callout>

## Errors


### ERC2981InvalidDefaultRoyalty


Thrown when default royalty fee exceeds 100% (10000 basis points).



```solidity
error ERC2981InvalidDefaultRoyalty(uint256 _numerator, uint256 _denominator);
```



### ERC2981InvalidDefaultRoyaltyReceiver


Thrown when default royalty receiver is the zero address.



```solidity
error ERC2981InvalidDefaultRoyaltyReceiver(address _receiver);
```



### ERC2981InvalidTokenRoyalty


Thrown when token-specific royalty fee exceeds 100% (10000 basis points).



```solidity
error ERC2981InvalidTokenRoyalty(uint256 _tokenId, uint256 _numerator, uint256 _denominator);
```



### ERC2981InvalidTokenRoyaltyReceiver


Thrown when token-specific royalty receiver is the zero address.



```solidity
error ERC2981InvalidTokenRoyaltyReceiver(uint256 _tokenId, address _receiver);
```

## Usage Example

```solidity
pragma solidity ^0.8.0;

library LibRoyalty {
    struct RoyaltyStorage {
        address receiver;
        uint96 feeNumerator;
        mapping(uint256 =&gt; RoyaltyInfo) tokenRoyalties;
    }

    struct RoyaltyInfo {
        address receiver;
        uint96 feeNumerator;
    }

    function getStorage() internal pure returns (RoyaltyStorage storage rs) {
        bytes32 slot = keccak256(&quot;diamond.storage.LibRoyalty&quot;);
        assembly {
            rs.slot := slot
        }
    }
}

contract DiamondExample {
    function setTokenRoyalty(uint256 _tokenId, address _receiver, uint96 _feeNumerator) external {
        LibRoyalty.RoyaltyStorage storage rs = LibRoyalty.getStorage();
        rs.tokenRoyalties[_tokenId] = LibRoyalty.RoyaltyInfo({receiver: _receiver, feeNumerator: _feeNumerator});
    }
    
    function royaltyInfo(uint256 _tokenId, uint256 _salePrice) external view returns (address, uint256) {
        LibRoyalty.RoyaltyStorage storage rs = LibRoyalty.getStorage();
        LibRoyalty.RoyaltyInfo memory info = rs.tokenRoyalties[_tokenId];
        if(info.receiver == address(0)) {
            return (rs.receiver, (_salePrice * rs.feeNumerator) / 10000);
        } else {
            return (info.receiver, (_salePrice * info.feeNumerator) / 10000);
        }
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Always validate receiver addresses and fee rates before calling royalty management functions to ensure the contract state remains consistent.
- Use `getStorage` to interact with the royalty storage struct. Do not duplicate storage handling logic in facets or external code.
- Optimize for future-proofing by defining sensible defaults for royalties and ensuring unused trailing variables can be removed in successive updates.
</Callout>



## Integration Notes

<Callout type="success" title="Shared Storage">
LibRoyalty operates through direct access to diamond storage, ensuring that royalty information updates are immediately visible across all facets that reference its storage slot. Use specific accessor methods like `getStorage` to maintain consistency and correctly implement EIP-8042.
</Callout>
