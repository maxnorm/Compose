<Callout type="info" title="Key Features">
- Implements the complete functionality of the ERC-1155 multi-token standard.
- Supports token-specific and batch URI resolution with &#x60;&amp;#123;id&amp;#125;&#x60; replacer syntax.
- Enables permissioned or permissionless token transfers with batched operations.
</Callout>

## Overview

The ERC1155Facet provides a modular implementation of the ERC-1155 multi-token standard for Compose diamonds. It facilitates token storage management, URI resolution, balance queries, approval settings, and secure token transfers. This facet is designed for highly extensible and composable multi-token systems without requiring standalone inheritance.

### uri

Returns the URI for token type &#x60;_id&#x60;. If a token-specific URI is set in tokenURIs[_id], returns the concatenation of baseURI and tokenURIs[_id]. Note that baseURI is empty by default and must be set explicitly if concatenation is desired. If no token-specific URI is set, returns the default URI which applies to all token types. The default URI may contain the substring &#x60;&amp;#123;id&amp;#125;&#x60; which clients should replace with the actual token ID.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {LibDiamond} from 'path/to/LibDiamond.sol';
import {ERC1155Facet} from 'path/to/ERC1155Facet.sol';

contract ExampleDiamond {
    function exampleUsage() external {
        address diamondProxy = address(this);
        ERC1155Facet erc1155 = ERC1155Facet(diamondProxy);

        // Query balance of a single token type
        uint256 balance = erc1155.balanceOf(msg.sender, 1);

        // Batch query balances
        address[] memory accounts = new address[](2);
        uint256[] memory ids = new uint256[](2);
        accounts[0] = msg.sender;
        accounts[1] = 0xAbc...;
        ids[0] = 1;
        ids[1] = 2;
        uint256[] memory balances = erc1155.balanceOfBatch(accounts, ids);

        // Approve another address for all token transfers
        erc1155.setApprovalForAll(0xDef..., true);

        // Transfer a single token type
        erc1155.safeTransferFrom(msg.sender, 0xAbc..., 1, 100, "0x");

        // Batch transfer multiple token types
        uint256[] memory tokenIds = new uint256[](2);
        uint256[] memory values = new uint256[](2);
        tokenIds[0] = 1;
        tokenIds[1] = 2;
        values[0] = 200;
        values[1] = 50;
        erc1155.safeBatchTransferFrom(msg.sender, 0xAbc..., tokenIds, values, "0x");
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Use &#x60;getStorage&#x60; for all storage reads and writes to maintain diamond compliance.
- Manage separate URI configuration for base URI and token-specific URIs via external initialization.
- Ensure consistent user approvals before invoking &#x60;safeTransferFrom&#x60; or &#x60;safeBatchTransferFrom&#x60;.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Validate user approvals in &#x60;setApprovalForAll&#x60;, &#x60;safeTransferFrom&#x60;, and &#x60;safeBatchTransferFrom&#x60; to prevent unauthorized token transfers. Use safe reentrancy patterns for token recipients and ensure input arrays in batched functions are aligned in length. Ensure URI fields do not unintentionally expose sensitive or invalid data.
</Callout>
