## Overview

The ERC20BurnFacet enables ERC20 token holders to destroy their tokens (burn) either from their own balance or from an allowance they have on another account. This functionality is crucial for reducing token supply in a controlled manner and integrates seamlessly within the Compose diamond proxy framework.


[View Source](https://github.com/maxnorm/Compose/blob/38545161423dd99e31b48b0eff41d260712fe731/src/token/ERC20/ERC20/ERC20BurnFacet.sol)



<Callout type="info" title="Key Features">
- Allows token holders to directly destroy (burn) their own tokens.
- Enables burning from allowances, ensuring controlled supply management.
- Emits `Transfer` events for transparent token destruction auditing.
</Callout>

## Usage Example

```solidity
pragma solidity ^0.8.0;

import { IDiamond } from &quot;./IDiamond.sol&quot;;

interface IERC20BurnFacet {
    function burn(uint256 _amount) external;
    function burnFrom(address _account, uint256 _amount) external;
}

contract Example {
    IDiamond diamond;

    constructor(address _diamond) {
        diamond = IDiamond(_diamond);
    }

    function reduceMyTokens(uint256 amount) external {
        IERC20BurnFacet(address(diamond)).burn(amount);
    }

    function reduceAnotherTokens(address account, uint256 amount) external {
        IERC20BurnFacet(address(diamond)).burnFrom(account, amount);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Always verify ERC20 allowances before calling `burnFrom` to prevent unexpected reverts.
- Use `burn` only when token destruction aligns with the tokenomics and governance rules.
- Integrate and test events such as `Transfer` to the zero address for proper accounting of burned tokens.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
Ensure proper access control and allowance checks to prevent unauthorized token burning. Test for reentrancy vulnerabilities, especially in the `burnFrom` function, as it interacts with allowances. Validate that the diamond storage slot for ERC20 is consistent and not overwritten by other facets in the diamond.
</Callout>
