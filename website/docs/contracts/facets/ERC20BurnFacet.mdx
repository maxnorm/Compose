Contract documentation for ERC20BurnFacet
</DocSubtitle>

<Callout type="info" title="Key Features">
- Provides secure token burning functionality from the caller&#x27;s balance or approved accounts.
- Utilizes the diamond storage pattern to maintain compatibility with other ERC-20 facets.
- Emits Transfer events to the zero address for transparent on-chain auditing.
</Callout>

## Overview

ERC20BurnFacet provides token destruction capabilities for ERC-20 token implementations within a Compose diamond. It supports burning tokens from the caller&#x27;s balance or another account with an approved allowance, and manages storage through the diamond&#x27;s predefined storage slot.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import { ERC20BurnFacet } from "./ERC20BurnFacet.sol";

contract UseDiamond {
    address diamondAddress;

    constructor(address _diamondAddress) {
        diamondAddress = _diamondAddress;
    }

    function burnTokens(uint256 amount) external {
        ERC20BurnFacet(diamondAddress).burn(amount);
    }

    function burnTokensFrom(address account, uint256 amount) external {
        ERC20BurnFacet(diamondAddress).burnFrom(account, amount);
    }

    function getTokenStorage() external view returns (bytes32 storageSlot) {
        return ERC20BurnFacet(diamondAddress).getStorage();
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure &#x60;getStorage&#x60; references the same slot as other ERC20 facets to avoid storage conflicts.
- Initialize token balances and allowances correctly through the appropriate initialization facet or process before enabling burn operations.
- For upgrades, verify storage consistency and backward compatibility with any new facets or changes.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Ensure access to the burnFrom function is restricted via allowances to prevent unauthorized token burns. Validate input amounts to prevent overflow or underflow issues. Avoid reentrancy risks by ensuring no external calls are made during state updates. Maintain proper coordination between all ERC-20 facets to avoid storage slot conflicts or unintended behaviors.
</Callout>
