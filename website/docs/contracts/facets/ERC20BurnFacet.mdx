## Overview

The ERC20BurnFacet provides functionality to reduce token supply by allowing users to burn their own balance or authorized balances from other accounts. It integrates seamlessly with diamond proxy contracts, offering modular and composable token burning capabilities.


[View Source](https://github.com/maxnorm/Compose/blob/7206f045d711880c211b967e80d72745b52f5cfc/src/token/ERC20/ERC20/ERC20BurnFacet.sol)



<Callout type="info" title="Key Features">
- Supports token burning from own balance or via allowance.
- Emits standardized Transfer events for burning operations.
- Utilizes diamond storage for modularity and efficiency.
</Callout>

## Errors


### ERC20InsufficientBalance


Thrown when an account has insufficient balance for a transfer or burn.



```solidity
error ERC20InsufficientBalance(address _sender, uint256 _balance, uint256 _needed);
```



### ERC20InsufficientAllowance


Thrown when a spender tries to use more than the approved allowance.




## Usage Example

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import { Diamond } from &quot;compose/Diamond.sol&quot;;
interface IERC20BurnFacet {
    function burn(uint256 _amount) external;
    function burnFrom(address _account, uint256 _amount) external;
}

contract ExampleUsage {
    address diamondAddress = &lt;diamond_address&gt;;
    
    function burnTokens(uint256 amount) external {
        IERC20BurnFacet facet = IERC20BurnFacet(diamondAddress);
        facet.burn(amount);
    }

    function burnAllowance(address account, uint256 amount) external {
        IERC20BurnFacet facet = IERC20BurnFacet(diamondAddress);
        facet.burnFrom(account, amount);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure sufficient balance and allowance before calling burn functions.
- Use custom errors to provide explicit failure reasons for transparency.
- Regularly audit the facet&#39;s integration to ensure compliance with the intended diamond storage structure.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
Ensure the diamond storage struct integrity when integrating facets to prevent overlapping state conflicts. Avoid reentrancy risks in hooks or callbacks linked to burn functionality. Properly limit access to burnFrom when managing allowances.
</Callout>
