Contract documentation for ERC20BurnFacet
</DocSubtitle>

<Callout type="info" title="Key Features">
- Supports direct token burns by the caller.
- Enables permissioned burns via delegated allowances.
- Emits standard Transfer events with the zero address to comply with ERC20 conventions.
</Callout>

## Overview

The ERC20BurnFacet provides functionality for token destruction, supporting both user-initiated burns and delegated burns via allowances. This facet operates on ERC20 storage within the diamond, emphasizing minimal state handling while emitting compliant Transfer events.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {LibDiamond} from 'path-to-LibDiamond.sol';
import {ERC20BurnFacet} from 'path-to-ERC20BurnFacet.sol';

contract MyDiamond {
    function exampleBurn(uint256 amount) external {
        ERC20BurnFacet(address(this)).burn(amount);
    }

    function exampleBurnFrom(address from, uint256 amount) external {
        ERC20BurnFacet(address(this)).burnFrom(from, amount);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Always initialize the ERC20 storage via associated initializer (outside this facet) before invoking burn functions.
- Verify the storage slot and struct compatibility during upgrades to ensure state coherency.
- Use only external calls and selector-based routing to access burn functions within the diamond.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Ensure input amounts and allowances are correctly validated to prevent underflow or overflow. Avoid storage slot manipulation outside designated patterns to maintain upgrade safety. Use reentrancy guards if exposing token burning to external systems that allow callback behaviors.
</Callout>
