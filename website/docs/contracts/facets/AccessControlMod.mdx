## Overview

The AccessControlMod facet provides role-based access control functionality for diamond contracts. It allows permission management by granting, revoking, and verifying roles, while enabling fine-grained access through an admin role hierarchy system. This is essential for modular contracts requiring secure and flexible authorization mechanisms.


[View Source](https://github.com/maxnorm/Compose/blob/7050baa250694d5e2445aa4c6b96c62c326cb1cb/src/access/AccessControl/AccessControlMod.sol)



<Callout type="info" title="Key Features">
- Role-based access control (RBAC) implementation.\n- Supports hierarchical role administration by assigning admin roles to other roles.\n- Allows permission checks with functions like hasRole and requireRole.
</Callout>

## Usage Example

```solidity
contract MyDiamond {\n    function initialize() external {\n        // Instance of AccessControlMod\n        AccessControlMod accessControl = AccessControlMod(address(this));\n\n        // Set &#39;ADMIN&#39; as the admin role for &#39;MODERATOR&#39;\n        bytes32 adminRole = keccak256(&quot;ADMIN&quot;);\n        bytes32 moderatorRole = keccak256(&quot;MODERATOR&quot;);\n        accessControl.setRoleAdmin(moderatorRole, adminRole);\n\n        // Grant &#39;ADMIN&#39; role to deployer\n        accessControl.grantRole(adminRole, msg.sender);\n\n        // Grant &#39;MODERATOR&#39; role to another account\n        address moderator = 0x1234567890abcdef1234567890abcdef12345678;\n        accessControl.grantRole(moderatorRole, moderator);\n    }\n\n    function restrictedFunction() external {\n        AccessControlMod accessControl = AccessControlMod(address(this));\n        bytes32 requiredRole = keccak256(&quot;MODERATOR&quot;);\n        accessControl.requireRole(requiredRole, msg.sender);\n\n        // Restricted logic here\n    }\n}
```

## Best Practices

<Callout type="tip" title="Best Practice">
- Always set an admin role for critical roles to enforce stricter controls.\n- Regularly audit and revoke unused or unnecessary roles to minimize attack vectors.\n- Ensure roles are only granted to trusted accounts and consider limiting their rights where appropriate.
</Callout>




## Security Considerations

<Callout type="warning" title="Security">
Ensure that only trusted roles and accounts can call functions like setRoleAdmin and grantRole, as improper use can compromise access controls. Always validate input data to prevent undesired access. Be cautious about reentrancy risks when role-related functions invoke external calls; utilize reentrancy guards where applicable.
</Callout>
