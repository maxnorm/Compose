## Overview

The RoyaltyFacet seamlessly integrates ERC-2981 compliant royalty management into a diamond contract. It allows retrieval and calculation of royalties either on a token-specific or default basis, ensuring compatibility with marketplace standards. This facet simplifies royalty operations while adhering to Compose's principle of modularity and reusability.


[View Source](https://github.com/maxnorm/Compose/blob/7050baa250694d5e2445aa4c6b96c62c326cb1cb/src/token/Royalty/RoyaltyFacet.sol)



<Callout type="info" title="Key Features">
- ERC-2981 compliant royalty calculation.\n- Token-specific or default royalty support.\n- Modular and reusable facet for seamless diamond integration.
</Callout>




## Functions


### getStorage


Returns a pointer to the royalty storage struct. Uses inline assembly to access the storage slot defined by STORAGE_POSITION.



```solidity
function getStorage() internal pure returns (RoyaltyStorage storage s);
```




**Returns**

| Name | Type | Description |
|------|------|-------------|

| `s` | `RoyaltyStorage` | The RoyaltyStorage struct in storage. |




### royaltyInfo


Returns royalty information for a given token and sale price. Returns token-specific royalty if set, otherwise falls back to default royalty. Royalty amount is calculated as a percentage of the sale price using basis points. Implements the ERC-2981 royaltyInfo function.



```solidity
function royaltyInfo(uint256 _tokenId, uint256 _salePrice)
    external
    view
    returns (address receiver, uint256 royaltyAmount);
```



**Parameters**

| Name | Type | Description |
|------|------|-------------|

| `_tokenId` | `uint256` | The NFT asset queried for royalty information. |

| `_salePrice` | `uint256` | The sale price of the NFT asset specified by _tokenId. |




**Returns**

| Name | Type | Description |
|------|------|-------------|

| `receiver` | `address` | The address designated to receive the royalty payment. |

| `royaltyAmount` | `uint256` | The royalty payment amount for _salePrice. |

## Usage Example

```solidity
pragma solidity ^0.8.0;\n\ninterface IDiamond {\n    function facetAddress(bytes4 selector) external view returns (address);\n}\n\ncontract UseRoyaltyFacet {\n    IDiamond constant diamond = IDiamond(0xYourDiamondAddress); // Replace with deployed diamond address.\n\n    function getRoyaltyInfo(uint256 tokenId, uint256 salePrice) external view returns (address, uint256) {\n        address royaltyFacet = diamond.facetAddress(bytes4(keccak256(&quot;royaltyInfo(uint256,uint256)&quot;)));\n        require(royaltyFacet != address(0), &quot;RoyaltyFacet not found&quot;);\n\n        (bool success, bytes memory result) = royaltyFacet.staticcall(abi.encodeWithSelector(\n            bytes4(keccak256(&quot;royaltyInfo(uint256,uint256)&quot;)),\n            tokenId,\n            salePrice\n        ));\n        require(success, &quot;Call to RoyaltyFacet failed&quot;);\n\n        return abi.decode(result, (address, uint256));\n    }\n}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the storage slot definition in `getStorage` aligns with the diamond's storage schema.\n- Avoid setting extremely high royalty percentages to maintain compatibility with marketplaces.\n- When modifying royalty settings, validate that changes do not inadvertently break ERC-2981 compliance.
</Callout>




## Security Considerations

<Callout type="warning" title="Security">
Carefully handle access control for functions that modify royalty storage to prevent unauthorized changes. In `royaltyInfo`, ensure proper input validation to prevent overflow attacks during royalty calculations. Be cautious with external calls in royalty management to avoid reentrancy vulnerabilities.
</Callout>
