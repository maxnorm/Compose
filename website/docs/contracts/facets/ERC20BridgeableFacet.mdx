<Callout type="info" title="Key Features">
- Provides cross-chain mint and burn functionality restricted to trusted bridges.
- Uses diamond storage pattern for efficient state management.
- Includes an internal check to validate bridge callers, ensuring robust access control.
</Callout>

## Overview

The ERC20BridgeableFacet provides cross-chain minting and burning functionality for ERC20 tokens within a Compose diamond. It ensures only trusted bridge addresses can perform these operations, leveraging diamond storage for both token and access control state management.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

interface IERC20BridgeableFacet {
    function crosschainMint(address _to, uint256 _amount) external;
    function crosschainBurn(address _from, uint256 _amount) external;
}

contract ExampleUsage {
    address diamondAddress = 0xYourDiamondAddress;

    function bridgeMint(address to, uint256 amount) external {
        IERC20BridgeableFacet(diamondAddress).crosschainMint(to, amount);
    }

    function bridgeBurn(address from, uint256 amount) external {
        IERC20BridgeableFacet(diamondAddress).crosschainBurn(from, amount);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Always ensure that the &#x60;trusted-bridge&#x60; role is correctly assigned to authorized addresses.
- Initialize access control and ERC20 storage explicitly during diamond deployment or upgrades.
- Test cross-chain mint and burn flows thoroughly to ensure compatibility with your bridge protocol.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Verify the integrity and trustworthiness of all addresses assigned the &#x60;trusted-bridge&#x60; role. Ensure proper input validation in crosschainMint and crosschainBurn to prevent malicious or unintended state changes. Avoid reentrancy risks by keeping external calls minimal or performing them after state updates.
</Callout>
