<Callout type="info" title="Key Features">
- Secure cross-chain mint and burn capabilities, gated by &#x60;trusted-bridge&#x60; roles.
- Built-in safeguards against unauthorized operations using role-based access control.
- Internal integrity check to evaluate and reject untrusted token bridge callers.
</Callout>

## Overview

The ERC20BridgeableFacet enables cross-chain token mint and burn functionality within a Compose diamond. It ensures that only trusted bridge addresses, verified via role-based access control, can mint or burn tokens across chains, supporting secure and modular cross-chain integrations.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {ERC20BridgeableFacet} from './ERC20BridgeableFacet.sol';

contract Example {
    function exampleUsage(address diamondAddress, address trustedBridge, address recipient, uint256 amount) external {
        // Mint tokens on the recipient's address via the trusted bridge
        ERC20BridgeableFacet(diamondAddress).crosschainMint(recipient, amount);

        // Burn tokens from the sender's address via the trusted bridge
        ERC20BridgeableFacet(diamondAddress).crosschainBurn(msg.sender, amount);
    }

    function checkTrustedBridge(address diamondAddress, address bridge) external view returns (bool) {
        // Verify if the given address has a trusted bridge role
        try ERC20BridgeableFacet(diamondAddress).checkTokenBridge(bridge) {
            return true;
        } catch {
            return false;
        }
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the AccessControl storage is correctly initialized during deployment to manage &#x60;trusted-bridge&#x60; roles.
- Validate ERC20 storage consistency across upgrades to avoid state-breaking changes.
- Limit cross-chain mint/burn calls to authorized operators by strictly controlling the role assignment during configuration or upgrades.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Ensure the &#x60;trusted-bridge&#x60; role assignments are tightly controlled and configured during deployment. Protect against reentrancy attacks by using the checks-effects-interactions pattern when cross-chain operations involve additional state changes. Validate inputs (e.g., non-zero recipient addresses, valid token amounts) to prevent unintended behavior. Only supply selectors of &#x60;external&#x60; functions directly accessible in the facet ABI; restructure or seal unused internals to prevent unintended exposure.
</Callout>
