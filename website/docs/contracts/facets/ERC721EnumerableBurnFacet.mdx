## Overview

The ERC721EnumerableBurnFacet allows tokens to be burned (permanently destroyed) while ensuring their removal from enumeration tracking, maintaining the integrity of token enumerability in a diamond contract. It provides a lightweight method to manage token lifecycles within ERC-721 implementations.


[View Source](https://github.com/maxnorm/Compose/blob/7206f045d711880c211b967e80d72745b52f5cfc/src/token/ERC721/ERC721Enumerable/ERC721EnumerableBurnFacet.sol)



<Callout type="info" title="Key Features">
- Burns ERC-721 tokens permanently while updating enumeration tracking.
- Provides access to facet-specific storage through a predictable and structured pattern.
</Callout>

## Usage Example

```solidity
pragma solidity ^0.8.0;

import { ERC721EnumerableBurnFacet } from &quot;./ERC721EnumerableBurnFacet.sol&quot;;

contract MyDiamondContract {
    function burnToken(uint256 _tokenId) external {
        // Assume access control is managed elsewhere in the diamond
        ERC721EnumerableBurnFacet burnFacet = ERC721EnumerableBurnFacet(address(this));
        burnFacet.burn(_tokenId);
    }

    function viewStorage() external view returns (bytes32) {
        ERC721EnumerableBurnFacet burnFacet = ERC721EnumerableBurnFacet(address(this));
        return burnFacet.getStorage();
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the burn function is restricted to authorized users, such as administrators or token owners.
- Properly test and verify the enumeration state before and after burning tokens.
- Clearly document how the burn process impacts the token ownership and enumeration mapping.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
Limit access to the `burn` function to ensure that only intended users (e.g., token owner, approved operator) can destroy tokens. Verify that the enumeration mappings are consistently updated and cannot be exploited to disrupt the sequence or create inconsistencies in the token list. Be cautious of reentrancy risks if external calls are introduced in future updates.
</Callout>
