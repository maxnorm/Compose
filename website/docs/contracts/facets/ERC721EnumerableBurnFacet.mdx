## Overview

The ERC721EnumerableBurnFacet provides functionality to burn (destroy) ERC-721 tokens, updating and maintaining enumeration state for token tracking. This facet is useful in diamond contracts for managing ERC-721 lifecycle and ensures accurate token enumeration after a burn operation.


[View Source](https://github.com/maxnorm/Compose/blob/7050baa250694d5e2445aa4c6b96c62c326cb1cb/src/token/ERC721/ERC721Enumerable/ERC721EnumerableBurnFacet.sol)



<Callout type="info" title="Key Features">
- Supports tracking of token enumeration state after burning\n- Getter for direct access to facet's storage\n- Facilitates token lifecycle management in composable diamond architectures
</Callout>

## Events

Emitted when ownership of a token changes, including burning.



```solidity

```



| Parameter | Type | Description |
|-----------|------|-------------|

| `_from` | `address` | The address transferring the token (or owner when burning). |

| `_to` | `address` | The address receiving the token (zero address when burning). |

| `_tokenId` | `uint256` | The ID of the token being transferred. |

## Best Practices

<Callout type="tip" title="Best Practice">
- Validate that the caller is authorized (e.g., token owner or approved user) before calling the `burn` function.\n- Regularly verify that enumeration data is consistent after burning tokens.\n- Ensure clear separation of concerns to avoid interfering with other facets handling ERC-721 tokens.
</Callout>




## Security Considerations

<Callout type="warning" title="Security">
Ensure proper access control to prevent unauthorized token burns. Validate ownership and approval status before invoking the burn function. Utilize reentrancy guards if interacting further with state changes post-burn.
</Callout>
