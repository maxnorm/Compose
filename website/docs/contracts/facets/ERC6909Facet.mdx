<Callout type="info" title="Key Features">
- Supports both fungible and non-fungible token semantics via a single ID.
- Provides granular control over token transfers and approvals.
- Enables operator roles for managing tokens on behalf of others.
</Callout>

## Overview

The ERC6909Facet implements the ERC-6909 standard, enabling fungible and non-fungible token management within a Compose diamond. It provides functions for querying balances, allowances, operator status, and performing transfers and approvals, acting as a core component for token functionalities.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC6909Facet} from "@compose-protocol/diamond-contracts/contracts/facets/erc6909/IERC6909Facet.sol";

contract ERC6909User {
    // Assume diamond contract is deployed and accessible
    address immutable DIAMOND_ADDRESS;

    constructor(address diamondAddress) {
        DIAMOND_ADDRESS = diamondAddress;
    }

    function getUserBalance(uint256 _id) external view returns (uint256) {
        // ERC6909Facet selector for balanceOf
        bytes4 selector = IERC6909Facet.balanceOf.selector;

        // Call balanceOf via diamond proxy
        (bool success, bytes memory data) = DIAMOND_ADDRESS.call(abi.encodeWithSelector(selector, address(this), _id));
        require(success, "ERC6909Facet: balanceOf call failed");
        return abi.decode(data, (uint256));
    }

    function transferTokens(uint256 _id, address _to, uint256 _amount) external {
        // ERC6909Facet selector for transfer
        bytes4 selector = IERC6909Facet.transfer.selector;

        // Call transfer via diamond proxy
        (bool success, ) = DIAMOND_ADDRESS.call(abi.encodeWithSelector(selector, _id, _to, _amount));
        require(success, "ERC6909Facet: transfer call failed");
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize the ERC6909Facet with appropriate access controls during diamond deployment.
- When interacting with the facet, always use the diamond proxy address.
- Design new storage structs to accommodate potential future trailing fields to maintain upgradeability.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Ensure robust access control mechanisms are implemented at the diamond level to restrict sensitive operations like setting operators or approving allowances to authorized addresses. Validate all input parameters, especially token IDs, amounts, sender, and receiver addresses, to prevent unexpected behavior or exploits. Reentrancy risks are mitigated by the diamond proxy pattern, but facet-level function logic should still be audited for potential reentrancy vectors if they call external contracts.
</Callout>
