## Overview

The ERC20Facet implements standard ERC-20 token functionality within a diamond contract architecture, allowing developers to manage token balances, approvals, and transfers seamlessly. It provides essential building blocks for creating composable and permissionless token systems within a diamond proxy framework.


[View Source](https://github.com/maxnorm/Compose/blob/e3e377cf3d77e94dc30e10812da2bdd1907ca159/src/token/ERC20/ERC20/ERC20Facet.sol)



<Callout type="info" title="Key Features">
- Implements core ERC-20 functions.
- Provides composable token design within diamond architecture.
- Emits standard events (Approval, Transfer) for transparency.
</Callout>

## Usage Example

```solidity
import &quot;contracts/diamond/ERC20Facet.sol&quot;;

contract MyDiamond {
    function transferTokens(address facetAddress, address to, uint256 amount) external {
        ERC20Facet(facetAddress).transfer(to, amount);
    }

    function checkBalance(address facetAddress, address user) external view returns (uint256) {
        return ERC20Facet(facetAddress).balanceOf(user);
    }

    function setApproval(address facetAddress, address spender, uint256 amount) external {
        ERC20Facet(facetAddress).approve(spender, amount);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Always validate token operations such as transfers and approvals to ensure compliance with expected rules.
- Minimize the use of `getStorage` externally unless necessary, as it is primarily intended for internal facet operations.
- Use clear and consistent event logging (e.g., Approval and Transfer) for easy audit trails.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
Ensure proper access control to prevent unauthorized approvals or transfers. Validate inputs to avoid integer overflows and underflows. Avoid reentrancy risks during external contract calls, especially in transfer-related functions.
</Callout>
