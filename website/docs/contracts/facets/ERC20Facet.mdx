## Overview

The ERC20Facet provides a modular implementation of ERC-20 token functions compatible with the Compose diamond standard. It enables seamless addition of standard token functionality, such as balance tracking, transfers, and approvals, to a diamond proxy contract.

[View Source](https://github.com/maxnorm/Compose/blob/bd92d896acdf08abd003a5846f267d6de9578ab5/src/token/ERC20/ERC20/ERC20Facet.sol)

<Callout type="info" title="Key Features">
- Provides full ERC-20 compliance within a Compose-compatible facet.\n- Supports standard token functions such as balances, approvals, and transfers.\n- Emits standard ERC-20 events (Approval, Transfer) for transparency and interoperability.
</Callout>

## Errors

### ERC20InsufficientBalance

Thrown when an account has insufficient balance for a transfer or burn.

```solidity
error ERC20InsufficientBalance(address _sender, uint256 _balance, uint256 _needed);
```

### ERC20InvalidSender

Thrown when the sender address is invalid (e.g., zero address).

```solidity
error ERC20InvalidSender(address _sender);
```

### ERC20InvalidReceiver

Thrown when the receiver address is invalid (e.g., zero address).

```solidity
error ERC20InvalidReceiver(address _receiver);
```

### ERC20InsufficientAllowance

Thrown when a spender tries to use more than the approved allowance.

```solidity
error ERC20InsufficientAllowance(address _spender, uint256 _allowance, uint256 _needed);
```

### ERC20InvalidSpender

Thrown when the spender address is invalid (e.g., zero address).

```solidity
error ERC20InvalidSpender(address _spender);
```

## Usage Example

```solidity
pragma solidity ^0.8.0;\n\nimport { Diamond } from &quot;./Diamond.sol&quot;;\n\ncontract ExampleDiamond {\n    Diamond private diamond;\n\n    constructor(address _diamond) {\n        diamond &#x3D; Diamond(_diamond);\n    }\n\n    function executeTransfer(address _to, uint256 _amount) external {\n        ERC20Facet(diamond).transfer(_to, _amount);\n    }\n\n    function checkBalance(address _account) external view returns (uint256) {\n        return ERC20Facet(diamond).balanceOf(_account);\n    }\n}
```

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure &#x60;getStorage&#x60; references the correct diamond storage slot to avoid overwriting unrelated data.\n- Use only as a fully self-contained facet without modifying its code to prevent unintended results.\n- Avoid mixing this facet with competing or overlapping token implementations within the same diamond.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Access control mechanisms should be implemented to prevent unauthorized calls to functions like approve. Ensure reentrancy protections are in place for state-changing functions, such as transfer and transferFrom, to avoid potential attacks. Review all facets added to the diamond for potential conflicts or overlaps in token management.
</Callout>
