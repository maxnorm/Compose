Contract documentation for ERC20PermitFacet
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements EIP-2612 for gasless token approvals via signatures.
- Provides a unique DOMAIN_SEPARATOR per chain ID to prevent cross-domain replay attacks.
- Tracks per-owner nonces to ensure each signature is used only once.
</Callout>

## Overview

The ERC20PermitFacet adds EIP-2612 permit functionality to ERC-20 tokens within a diamond proxy. It allows token allowances to be set via signatures without requiring direct on-chain transactions, reducing gas costs and enhancing usability for meta-transactions or off-chain approvals.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {Diamond} from "path/to/diamond.sol";
import {ERC20PermitFacet} from "path/to/ERC20PermitFacet.sol";

contract Example {
    function configureDiamond() external {
        Diamond diamond = new Diamond();
        ERC20PermitFacet permitFacet = new ERC20PermitFacet();

        // Add PermitFacet to the diamond
        bytes4[] memory selectors = new bytes4[](5);
        selectors[0] = permitFacet.getERC20Storage.selector;
        selectors[1] = permitFacet.getStorage.selector;
        selectors[2] = permitFacet.nonces.selector;
        selectors[3] = permitFacet.DOMAIN_SEPARATOR.selector;
        selectors[4] = permitFacet.permit.selector;

        diamond.addFacet(address(permitFacet), selectors);
    }

    function usePermitFacet(Diamond diamond, address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) external {
        ERC20PermitFacet permitFacet = ERC20PermitFacet(diamond);
        permitFacet.permit(owner, spender, value, deadline, v, r, s);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize DOMAIN_SEPARATOR once on deployment using the chain ID.
- Ensure users are aware of nonce mechanics to prevent mismatches or signature reuse.
- Use only trusted external libraries for signature validation to ensure correctness and security.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Verify signature inputs (e.g., v, r, s) for validity and prevent reentrancy or manipulation during state updates. Ensure DOMAIN_SEPARATOR initialization is deterministic using the correct chain ID. Protect against replay attacks by using nonces and DOMAIN_SEPARATOR correctly. Verify spender and value parameters to prevent invalid allowances.
</Callout>
