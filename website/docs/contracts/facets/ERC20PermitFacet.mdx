## Overview

The ERC20PermitFacet adds EIP-2612 permit functionality to a diamond contract, enabling token holders to grant approvals via off-chain signatures instead of on-chain transactions. This eliminates the need for separate approval transactions, reducing gas costs and improving usability in complex dApp interactions.


[View Source](https://github.com/maxnorm/Compose/blob/eabc77b81e461d751a5a8e003c3a9c2e818b9d3c/src/token/ERC20/ERC20Permit/ERC20PermitFacet.sol)



<Callout type="info" title="Key Features">
- Implements EIP-2612 permit functionality to allow gasless approval.\n- Maintains compatibility with existing ERC-20 token standards.\n- Simplifies token approvals in meta-transaction schemes.
</Callout>

### permit


Sets the allowance for a spender via a signature. This function implements EIP-2612 permit functionality.



```solidity
function permit(
    address _owner,
    address _spender,
    uint256 _value,
    uint256 _deadline,
    uint8 _v,
    bytes32 _r,
    bytes32 _s
) external;
```



**Parameters**

| Name | Type | Description |
|------|------|-------------|

| `_owner` | `address` | The address of the token owner. |

| `_spender` | `address` | The address of the spender. |

| `_value` | `uint256` | The amount of tokens to approve. |

| `_deadline` | `uint256` | The deadline for the permit (timestamp). |

| `_v` | `uint8` | The recovery byte of the signature. |

| `_r` | `bytes32` | The r value of the signature. |

| `_s` | `bytes32` | The s value of the signature. |

## Usage Example

```solidity
pragma solidity ^0.8.0;\n\ninterface IDiamond {\n    function facetAddress(bytes4 selector) external view returns (address);\n}\n\ncontract Example {\n    IDiamond diamond;\n\n    constructor(address _diamond) {\n        diamond = IDiamond(_diamond);\n    }\n\n    function permitSpend(address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) external {\n        address facet = diamond.facetAddress(bytes4(keccak256(&quot;permit(address,address,uint256,uint256,uint8,bytes32,bytes32)&quot;)));\n        require(facet != address(0), &quot;ERC20PermitFacet not found&quot;);\n        (bool success, ) = facet.delegatecall(\n            abi.encodeWithSignature(\n                &quot;permit(address,address,uint256,uint256,uint8,bytes32,bytes32)&quot;,\n                owner,\n                spender,\n                value,\n                deadline,\n                v,\n                r,\n                s\n            )\n        );\n        require(success, &quot;Permit failed&quot;);\n    }\n}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Always verify that the DOMAIN_SEPARATOR is correct and matches the chain ID to prevent signature mismatches.\n- Validate the nonce before calling permit to ensure that the signature corresponds to the correct transaction state.\n- Use an audited and correctly configured facet to integrate EIP-2612 securely.
</Callout>




## Security Considerations

<Callout type="warning" title="Security">
The permit function must validate the provided signature against the DOMAIN_SEPARATOR to prevent replay attacks across chains. Be cautious of reentrancy risks and ensure proper access control measures are in place when integrating this functionality. Only rely on well-formed nonces to prevent unauthorized signature reuse.
</Callout>
