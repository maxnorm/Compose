## Overview

The ERC20PermitFacet implements EIP-2612, enabling token allowances to be set via signatures instead of on-chain transactions. This improves user experience by saving gas and allowing approvals to be made off-chain, making it especially useful in scenarios requiring meta-transactions or interacting with token contracts in a gas-optimized manner.


[View Source](https://github.com/maxnorm/Compose/blob/e3e377cf3d77e94dc30e10812da2bdd1907ca159/src/token/ERC20/ERC20/ERC20PermitFacet.sol)



<Callout type="info" title="Key Features">
- Implements EIP-2612 permit functionality
- Supports gasless approvals via off-chain signatures
- Provides domain separator and nonce tracking for security and replay protection
</Callout>

## Usage Example

```solidity
pragma solidity ^0.8.17;

import {IDiamondCut} from &quot;./interfaces/IDiamondCut.sol&quot;;
import {Diamond} from &quot;./Diamond.sol&quot;;

contract ExampleDiamond {
    Diamond public diamond;

    constructor() {
        diamond = new Diamond(address(this));
    }

    function setAllowanceWithPermit(
        address owner,
        address spender,
        uint256 value,
        uint256 deadline,
        uint8 v,
        bytes32 r,
        bytes32 s
    ) external {
        // Call the permit function on the ERC20 facet of the diamond
        ERC20PermitFacet(address(diamond)).permit(owner, spender, value, deadline, v, r, s);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure valid signatures are verified using right combinations of `v`, `r`, and `s`.
- Always validate `deadline` to protect against expired permits.
- Use the `DOMAIN_SEPARATOR` to prevent replay attacks across chains or contracts.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
When using the `permit` function, ensure the signature corresponds exactly to the expected owner and payload. Validate the nonce and `DOMAIN_SEPARATOR` to avoid replay attacks. Operations relying on `permit` should guard against re-entrancy issues, especially if interacting with third-party contracts after setting allowances.
</Callout>
