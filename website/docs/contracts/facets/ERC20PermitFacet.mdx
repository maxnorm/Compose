## Overview

The ERC20PermitFacet enables gasless approval of token allowances using EIP-2612 standard permits. It incorporates signature-based authorization via `permit`, with nonce and domain separator management to ensure security and prevent replay attacks. This facet reduces reliance on explicit transaction approvals, improving UX in diamond proxy systems.


[View Source](https://github.com/maxnorm/Compose/blob/7206f045d711880c211b967e80d72745b52f5cfc/src/token/ERC20/ERC20/ERC20PermitFacet.sol)



<Callout type="info" title="Key Features">
- Enables gasless token approval.
- Incorporates EIP-2612 standard security against replay attacks.
- Provides `nonces` and `DOMAIN_SEPARATOR` for efficient integration.
</Callout>

## Usage Example

```solidity
pragma solidity ^0.8.0;

import { IDiamondCut } from &quot;diamond-standard/interfaces/IDiamondCut.sol&quot;;

contract UseERC20Permit {
    function executePermit(
        address diamond,
        address owner,
        address spender,
        uint256 value,
        uint256 deadline,
        uint8 v,
        bytes32 r,
        bytes32 s
    ) external {
        // Call the permit function in the ERC20PermitFacet within a diamond contract
        (bool success, bytes memory data) = diamond.call(
            abi.encodeWithSignature(
                &quot;permit(address,address,uint256,uint256,uint8,bytes32,bytes32)&quot;,
                owner,
                spender,
                value,
                deadline,
                v,
                r,
                s
            )
        );
        require(success, &quot;Permit execution failed&quot;);
    }
}
```



## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `permit` function is used only for EIP-2612 compliant tokens.
- Verify the returned `nonce` matches your expected value before and after permitting.
- Securely store or validate the DOMAIN_SEPARATOR to prevent incorrect signature passing.
</Callout>



## Security Considerations

<Callout type="warning" title="Security">
Verify the signature validity and ensure `DOMAIN_SEPARATOR` aligns with the diamond&#39;s contract state. Handle replay attacks by tracking and synchronizing `nonces` correctly. Use access control mechanisms to prevent misuse of other inherited functions in the diamond.
</Callout>
