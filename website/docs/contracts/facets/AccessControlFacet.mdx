Contract documentation for AccessControlFacet
</DocSubtitle>

<Callout type="info" title="Key Features">
- Role-based permissioning with hierarchical admin roles for governance.
- Batch operations for granting and revoking roles, improving gas efficiency.
- Fine-grained access control with safeguards like &#x60;requireRole&#x60; for conditional execution.
</Callout>

## Overview

The AccessControlFacet provides fine-grained role-based access control mechanisms within a Compose diamond proxy. It supports role management, including assignment, revocation, hierarchical admin role permissions, and batch operations, enabling modular and secure permissioning for decentralized applications.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

interface IAccessControlFacet {
    function grantRole(bytes32 role, address account) external;
    function revokeRole(bytes32 role, address account) external;
    function hasRole(bytes32 role, address account) external view returns (bool);
}

contract MyContract {
    address diamondAddress;

    // Replace with actual role identifiers
    bytes32 constant ROLE_ADMIN = keccak256("ROLE_ADMIN");
    bytes32 constant ROLE_USER = keccak256("ROLE_USER");

    constructor(address _diamondAddress) {
        diamondAddress = _diamondAddress;
    }

    function setupRoles(address admin, address user) external {
        IAccessControlFacet(diamondAddress).grantRole(ROLE_ADMIN, admin);
        IAccessControlFacet(diamondAddress).grantRole(ROLE_USER, user);
    }

    function removeUserRole(address user) external {
        IAccessControlFacet(diamondAddress).revokeRole(ROLE_USER, user);
    }

    function checkUserRole(address user) external view returns (bool) {
        return IAccessControlFacet(diamondAddress).hasRole(ROLE_USER, user);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Always initialize roles and the role hierarchy (assigning admin roles) properly during deployment or initialization.
- Restrict access to role management functions using a clear and secure role hierarchy.
- Test role assumptions under various edge cases, especially for batch operations, to ensure no unintended permission escalation.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Ensure role management functions, like &#x60;grantRole&#x60; and &#x60;setRoleAdmin&#x60;, are restricted to appropriate admin roles using robust access checks. Validate all role identifiers and account inputs to mitigate typos or attacks. Be cautious with the &#x60;grantRoleBatch&#x60; and &#x60;revokeRoleBatch&#x60; functions to prevent over-permissioning or accidental revocation due to incorrect arrays. Avoid circular admin role dependencies to prevent breaking the permission hierarchy.
</Callout>
