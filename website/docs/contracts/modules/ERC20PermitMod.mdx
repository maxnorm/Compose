LibERC20Permit — Library for ERC-2612 Permit Logic - Thrown when a permit signature is invalid or expired.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements ERC-2612 permit functionality, enabling off-chain signed allowance grants.
- Provides &#x60;DOMAIN_SEPARATOR&#x60; for correct signature encoding and replay protection.
- Integrates seamlessly with ERC20Storage and PermitStorage within the diamond.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

The ERC20PermitMod library provides essential logic for implementing the ERC-2612 permit functionality within a Compose diamond. It enables users to grant allowances via signed off-chain messages, enhancing gas efficiency and user experience for ERC-20 token interactions. This module is crucial for any diamond that requires delegated spending capabilities for its ERC-20 tokens.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC20PermitMod} from "@compose/modules/ERC20PermitMod.sol";
import {ERC20Storage} from "@compose/storage/ERC20Storage.sol";
import {PermitStorage} from "@compose/storage/PermitStorage.sol";

contract MyERC20Facet {
    using ERC20Storage for ERC20Storage.Layout;
    using PermitStorage for PermitStorage.Layout;
    using IERC20PermitMod for IERC20PermitMod.Layout;

    IERC20PermitMod.Layout immutable _erc20PermitMod;

    constructor(address _diamondAddress) {
        _erc20PermitMod = IERC20PermitMod.Layout.layout(_diamondAddress);
    }

    /**
     * @notice Allows a user to grant allowance via a signed permit.
     * @param owner The address of the owner granting the allowance.
     * @param spender The address to which the allowance is granted.
     * @param value The amount of tokens being allowed.
     * @param deadline The timestamp after which the permit is no longer valid.
     * @param v The v component of the signature.
     * @param r The r component of the signature.
     * @param s The s component of the signature.
     */
    function permit(address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) external {
        _erc20PermitMod.permit(owner, spender, value, deadline, v, r, s);
    }

    /**
     * @notice Retrieves the domain separator for permit signatures.
     * @return The domain separator.
     */
    function DOMAIN_SEPARATOR() external view returns (bytes32) {
        return _erc20PermitMod.DOMAIN_SEPARATOR();
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the &#x60;permit&#x60; function is called with valid signature parameters (v, r, s) and a non-expired deadline to prevent signature validation failures.
- Implement proper access control for functions that might indirectly interact with permit allowances, although &#x60;permit&#x60; itself is typically permissionless.
- Be aware that the &#x60;permit&#x60; function emits an &#x60;Approval&#x60; event, which should be handled by the calling facet or consumer.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
The ERC20PermitMod library relies on the standard Compose diamond storage pattern. It accesses and modifies state within the &#x60;ERC20Storage&#x60; and &#x60;PermitStorage&#x60; structs, specifically updating allowances. The &#x60;permit&#x60; function must be implemented by a facet that has access to these storage layouts. The function validates signatures against the domain separator and the chain ID, ensuring that permits are chain-specific and cannot be replayed across different networks. The &#x60;Approval&#x60; event is emitted by the &#x60;permit&#x60; function itself, and it is the responsibility of the facet implementing it to ensure this event is correctly propagated.
</Callout>
