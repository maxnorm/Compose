LibERC20Permit — Library for ERC-2612 Permit Logic - Thrown when a permit signature is invalid or expired.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Gasless, off-chain approval mechanism via valid signatures.
- Enforces chain-specific replay protection with a unique DOMAIN_SEPARATOR per contract and chain ID.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

ERC20PermitMod provides ERC-2612 permit functionality for Compose diamonds, enabling gasless approvals without requiring direct interactions with the token holder&#x27;s wallet. This improves composability and safety by decentralizing signature validation while leveraging diamond storage for the ERC20 and Permit logic.

    function permit(address _owner, address _spender, uint256 _value, uint256 _deadline, uint8 _v, bytes32 _r, bytes32 _s) external;

    function approveViaPermit(address _token, address _spender, uint256 _value, uint256 _deadline, uint8 _v, bytes32 _r, bytes32 _s) external {
        IERC20PermitFacet(_token).permit(msg.sender, _spender, _value, _deadline, _v, _r, _s);
    }
}`}

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure DOMAIN_SEPARATOR is recomputed if the contract is chain-migrated to avoid replay attacks.
- Emit an Approval event whenever the permit function updates allowances.
- Validate EIP-2612 parameters in facets calling &#x60;permit&#x60; to avoid misuse or signature forgery.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
ERC20PermitMod relies on diamond storage patterns for state isolation and safe upgrades. Changes made through the &#x60;permit&#x60; function are only visible within a diamond that integrates the relevant facets, ensuring modularity. Ensure facets explicitly handle the approval event emission post-validation to maintain compatibility with ERC-20 standards.
</Callout>
