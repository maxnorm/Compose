<DocSubtitle>
Emitted when the admin role for a role is changed.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Role-based access control for granular permission management.
- Functions to grant, revoke, and check for specific roles assigned to accounts.
- Ability to set and manage admin roles for other roles, creating a hierarchy.
- Built-in error &#x60;AccessControlUnauthorizedAccount&#x60; for failed role checks.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

The AccessControlMod module provides a robust system for managing role-based access control within a Compose diamond. It enables granular permissioning by allowing administrators to grant, revoke, and check roles for accounts, ensuring that sensitive functions are only accessible to authorized entities. This modular approach simplifies access control logic and enhances the security posture of the diamond.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IAccessControlMod} from "@compose/modules/AccessControlMod.sol";

contract MyFacet {
    IAccessControlMod private accessControl;

    constructor(address _diamondProxy) {
        accessControl = IAccessControlMod(_diamondProxy);
    }

    /**
     * @notice Grants a role to an account.
     */
    function grantMyRole(address _account, bytes32 _role) external {
        // Assuming 'MyRole' is a role managed by this facet
        accessControl.grantRole(_role, _account);
    }

    /**
     * @notice Checks if an account has a specific role.
     */
    function checkMyRole(address _account, bytes32 _role) external view returns (bool) {
        return accessControl.hasRole(_role, _account);
    }

    /**
     * @notice Revokes a role from an account.
     */
    function revokeMyRole(address _account, bytes32 _role) external {
        accessControl.revokeRole(_role, _account);
    }
}
`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Use &#x60;requireRole&#x60; for explicit access control checks at the beginning of functions to enforce permissions.
- Store role identifiers (&#x60;bytes32&#x60;) in constants or enums for clarity and to prevent typos.
- Ensure that the admin role for sensitive roles is appropriately managed, potentially by a dedicated admin contract or a multi-sig.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
The AccessControlMod utilizes a dedicated storage slot within the diamond proxy to maintain its role mappings. Facets interact with this module by casting the diamond proxy address to the &#x60;IAccessControlMod&#x60; interface. Any changes made to roles or role admins via &#x60;grantRole&#x60;, &#x60;revokeRole&#x60;, or &#x60;setRoleAdmin&#x60; are immediately reflected across all facets interacting with the AccessControl module through the diamond. The module does not impose specific ordering requirements for storage, but it&#x27;s crucial to ensure its storage slot is not overwritten by other modules.
</Callout>
