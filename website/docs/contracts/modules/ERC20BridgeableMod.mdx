## Overview

The ERC20BridgeableMod provides functionality essential for cross-chain token operations within a diamond proxy. It securely facilitates minting and burning through trusted bridges while ensuring proper access control and storage compatibility. This module is critical for enabling seamless token interoperability and maintaining on-chain integrity.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

interface IERC20Bridgeable {
    function crosschainMint(address _receiver, uint256 _amount) external;
    function crosschainBurn(address _holder, uint256 _amount) external;
}

contract MyCustomFacet {
    address private constant diamond = 0xYourDiamondAddress;

    /**
     * Cross-mint tokens to a receiver via a trusted bridge.
     */
    function mintTokens(address _receiver, uint256 _amount) external {
        IERC20Bridgeable(diamond).crosschainMint(_receiver, _amount);
    }

    /**
     * Execute a cross-chain burn triggered by a bridge.
     */
    function burnTokens(address _holder, uint256 _amount) external {
        IERC20Bridgeable(diamond).crosschainBurn(_holder, _amount);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Validate that callers are assigned the &#x60;trusted-bridge&#x60; role before initiating crosschainMint or crosschainBurn to prevent unauthorized actions.
- Ensure the correct storage slots are referenced by maintaining consistent and verified diamond storage layouts.
- Handle potential reverts from invalid zero address or untrusted bridge checks to maintain integrity during token operations.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
The module interacts with diamond storage to retrieve and use the AccessControl and ERC20 storage structs, ensuring consistent data management across facets. Functions like getAccessControlStorage and getERC20Storage leverage inline assembly to directly reference predefined storage slots, which must remain unchanged for proper operation. Changes and updates to the module or storage structure should follow Compose&#x27;s storage reuse guidelines to preserve compatibility and safety. Ensure the trusted roles are properly assigned before initiating cross-chain actions for state consistency.
</Callout>
