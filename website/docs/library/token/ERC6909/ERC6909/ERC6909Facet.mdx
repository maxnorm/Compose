<DocSubtitle>
Manage ERC-6909 token balances and operator permissions.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements ERC-6909 standard for fungible and non-fungible tokens.
- Supports token transfers, balance queries, and allowance management.
- Enables setting and checking operator permissions for token management.
</Callout>

## Overview

This facet implements the ERC-6909 standard, enabling token transfers, balance checks, and operator management within a Compose diamond. It provides essential functionality for managing fungible and non-fungible assets, including allowances and operator relationships.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC6909Facet} from "@compose/contracts/facets/ERC6909/IERC6909Facet.sol";

contract ERC6909Consumer {
    IERC6909Facet public immutable erc6909Facet;

    constructor(address _erc6909FacetAddress) {
        erc6909Facet = IERC6909Facet(_erc6909FacetAddress);
    }

    function consumeTransfer(address _to, uint256 _amount, bytes32 _id) external {
        // Assuming token ID is represented by bytes32
        erc6909Facet.transfer(_to, _amount, _id);
    }

    function consumeApprove(address _spender, uint256 _amount, bytes32 _id) external {
        erc6909Facet.approve(_spender, _amount, _id);
    }

    function consumeSetOperator(address _operator, bool _approved, bytes32 _id) external {
        erc6909Facet.setOperator(_operator, _approved, _id);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `ERC6909Facet` is correctly initialized with the diamond proxy.
- Implement appropriate access control for functions that modify operator status or are critical for specific token types.
- When adding new storage for ERC6909, ensure it follows the Compose storage pattern to maintain upgradeability.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The `transfer` and `transferFrom` functions should be carefully protected by access control mechanisms if the tokens managed are not fully permissionless. Ensure that `approve` and `setOperator` calls are validated to prevent unintended approvals or operator assignments. Reentrancy is mitigated by the standard diamond proxy pattern, but facet-specific logic should be reviewed for potential vulnerabilities.
</Callout>
