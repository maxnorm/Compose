<DocSubtitle>
Implements ERC-20 token functionality within a diamond
</DocSubtitle>

<Callout type="info" title="Key Features">
- Exposes standard ERC-20 functions (transfer, approve, balanceOf, etc.) via the diamond proxy.
- Manages token state (totalSupply, balances, allowances) using the diamond storage pattern.
- Emits ERC-20 standard `Transfer` and `Approval` events.
- Self-contained, adhering to Compose facet design principles.
</Callout>

## Overview

This facet implements core ERC-20 token functions such as transfers, approvals, and balance queries within a diamond proxy. It accesses shared token state via the diamond storage pattern, enabling composability and upgradeability for token operations. Developers integrate this facet to expose standard ERC-20 behavior through the diamond's unified interface.

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize token metadata (name, symbol, decimals) and total supply during diamond deployment.
- Ensure appropriate access controls are applied to functions that modify token state if required by the diamond's architecture.
- Verify storage compatibility before upgrading to a new version of this facet to prevent state corruption.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Functions like `transfer`, `transferFrom`, and `approve` implement standard ERC-20 checks to prevent insufficient balance or allowance issues, reverting with custom errors (`ERC20InsufficientBalance`, `ERC20InsufficientAllowance`). Input validation is performed on addresses to ensure they are not the zero address for relevant operations. Follow standard Solidity security practices for input validation and reentrancy guards if further customization is applied.
</Callout>
