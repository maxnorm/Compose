<DocSubtitle>
Burns ERC-20 tokens from caller or other accounts
</DocSubtitle>

<Callout type="info" title="Key Features">
- Exposes external functions for diamond routing of token burning.
- Self-contained facet with no external dependencies or inheritance.
- Supports burning tokens from the caller's balance or from another specified account.
- Utilizes custom errors for efficient error handling.
</Callout>

## Overview

This facet implements ERC-20 token burning functionality within a diamond. It exposes external functions to destroy tokens from the caller's balance or from another specified account. Calls are routed through the diamond proxy, accessing shared token storage. Developers integrate this facet to enable token destruction mechanisms while maintaining the diamond's upgradeability.

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `ERC20BurnFacet` is correctly added to the diamond's facet registry during deployment.
- Grant necessary permissions if `burnFrom` requires specific access control mechanisms beyond token allowances.
- Verify that the diamond's storage layout is compatible with the `ERC20Storage` struct before upgrading.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The `burn` function burns tokens from the caller's balance. The `burnFrom` function burns tokens from a specified account's balance, requiring the caller to have sufficient allowance. Both functions emit a `Transfer` event to the zero address. Input validation is handled by Solidity's built-in checks and the custom `ERC20InsufficientBalance` and `ERC20InsufficientAllowance` errors. Follow standard Solidity security practices.
</Callout>
