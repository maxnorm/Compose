<DocSubtitle>
Enables EIP-2612 permit functionality for ERC-20 tokens.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements EIP-2612 permit for off-chain approvals.
- Utilizes `nonces` to prevent signature replay attacks.
- Provides `DOMAIN_SEPARATOR` for correct EIP-712 message hashing.
</Callout>

## Overview

The ERC20PermitFacet integrates EIP-2612's permit functionality into a Compose diamond, allowing users to grant token allowances via signed messages. This enhances gas efficiency and user experience by enabling off-chain approvals that can be later executed on-chain.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC20Permit, ERC20PermitFacet} from "@compose/contracts/src/facets/ERC20Permit/ERC20PermitFacet.sol";
import {DiamondInterface} from "@compose/contracts/src/interfaces/DiamondInterface.sol";

contract ERC20PermitUser {
    DiamondInterface immutable diamond;

    constructor(address _diamondAddress) {
        diamond = DiamondInterface(_diamondAddress);
    }

    function permitToken(address _token, address _spender, uint256 _value, uint256 _deadline, bytes calldata _signature) external {
        // Assuming ERC20PermitFacet is registered under a known selector
        address erc20PermitFacetAddress = diamond.facetAddress(diamond.facetFunctionSelector(ERC20PermitFacet.permit.selector));
        IERC20Permit(erc20PermitFacetAddress).permit(_token, msg.sender, _spender, _value, _deadline, _signature);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize the ERC20PermitFacet with the correct ERC-20 token contracts it will manage.
- Ensure the signature provided to the `permit` function is valid and corresponds to the owner's EIP-712-encoded message.
- Store the `DOMAIN_SEPARATOR` and `nonces` for signature verification and replay protection.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The `permit` function is permissionless and relies on the validity of the provided signature. Ensure robust signature verification mechanisms are in place. The `nonces` mapping is critical for preventing replay attacks; its state must be managed correctly. The facet does not perform spender validation, relying on the underlying ERC-20 token's `approve` function or equivalent logic executed by the permit caller.
</Callout>
