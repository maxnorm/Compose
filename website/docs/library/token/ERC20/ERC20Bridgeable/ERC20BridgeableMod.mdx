<DocSubtitle>
Manages cross-chain ERC20 token bridging operations.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Restricts minting and burning operations to addresses holding the `trusted-bridge` role.
- Provides internal checks for validating bridge accounts and recipient/sender addresses.
- Leverages diamond storage for managing bridge-related state and access control configurations.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

The ERC20BridgeableMod provides essential functionality for cross-chain ERC20 token transfers. It ensures that only authorized bridge accounts can perform minting and burning operations across different chains, enhancing security and control for bridged assets. This module integrates seamlessly with Compose diamonds by leveraging the diamond storage pattern for robust state management.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC20BridgeableMod} from "@compose-protocol/diamond-contracts/contracts/modules/ERC20Bridgeable/IERC20BridgeableMod.sol";
import {AccessControlErrors} from "@compose-protocol/diamond-contracts/contracts/modules/AccessControl/AccessControl.sol";

contract MyBridgeFacet {
    IERC20BridgeableMod public immutable erc20BridgeableMod;

    constructor(address _diamondAddress) {
        erc20BridgeableMod = IERC20BridgeableMod(_diamondAddress);
    }

    /**
     * @notice Burns an ERC20 token on the current chain for cross-chain transfer.
     * @param _token Address of the ERC20 token to burn.
     * @param _to The recipient address on the destination chain.
     * @param _amount The amount of tokens to burn.
     */
    function burnForCrosschain(address _token, address _to, uint256 _amount) external {
        // Assume caller is a trusted bridge address, enforced by the module.
        erc20BridgeableMod.crosschainBurn(_token, _to, _amount);
    }

    /**
     * @notice Mints an ERC20 token on the current chain as part of a cross-chain transfer.
     * @param _token Address of the ERC20 token to mint.
     * @param _to The recipient address on the destination chain.
     * @param _amount The amount of tokens to mint.
     */
    function mintFromCrosschain(address _token, address _to, uint256 _amount) external {
        // Assume caller is a trusted bridge address, enforced by the module.
        erc20BridgeableMod.crosschainMint(_token, _to, _amount);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure that only addresses with the `trusted-bridge` role can call `crosschainBurn` and `crosschainMint` functions. The module enforces this internally.
- Handle `ERC20InvalidBridgeAccount`, `ERC20InvalidCallerAddress`, `ERC20InvalidReciever`, and `ERC20InvalidSender` errors appropriately to manage invalid bridge operations.
- Verify that the `_token` address passed to bridging functions is a valid ERC20 token contract.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
This module interacts with the diamond's storage for AccessControl and ERC20 state. The `getAccessControlStorage` and `getERC20Storage` functions provide direct access to these storage structs via inline assembly. Callers must ensure they are interacting with a deployed diamond that has these modules correctly initialized. The module relies on the `trusted-bridge` role defined within the AccessControl facet. Changes to the `trusted-bridge` role will directly impact the operability of cross-chain functions.
</Callout>
