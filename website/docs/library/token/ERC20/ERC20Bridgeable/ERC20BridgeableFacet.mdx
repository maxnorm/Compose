<DocSubtitle>
Facilitates cross-chain ERC20 token operations.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Permissioned cross-chain minting and burning of ERC20 tokens.
- Enforces `trusted-bridge` role for all cross-chain operations.
- Provides access to internal storage structs for configuration inspection.
</Callout>

## Overview

The ERC20BridgeableFacet enables secure and authorized cross-chain minting and burning of ERC20 tokens. It enforces access control for trusted bridge operators and provides utility functions to inspect token and access control configurations.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC20BridgeableFacet} from "@compose/contracts/src/facets/ERC20Bridgeable/IERC20BridgeableFacet.sol";

contract ERC20BridgeableConsumer {
    IERC20BridgeableFacet public erc20BridgeableFacet;

    constructor(address _diamondAddress) {
        erc20BridgeableFacet = IERC20BridgeableFacet(_diamondAddress);
    }

    function mintOnRemoteChain(address _token, address _to, uint256 _amount) external {
        // Assuming _token is an address of an ERC20 compliant token that is bridgeable
        // This function would be called by a trusted bridge operator
        erc20BridgeableFacet.crosschainMint(_token, _to, _amount);
    }

    function burnOnRemoteChain(address _token, address _from, uint256 _amount) external {
        // This function would be called by a trusted bridge operator
        erc20BridgeableFacet.crosschainBurn(_token, _from, _amount);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `trusted-bridge` role is granted only to verified and secure bridge operator addresses.
- Utilize `getERC20Storage` and `getAccessControlStorage` for introspection of facet configurations.
- Implement robust off-chain or guardian mechanisms to coordinate cross-chain transfers and verify mint/burn operations.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Access to `crosschainMint` and `crosschainBurn` is restricted to addresses holding the `trusted-bridge` role. The `checkTokenBridge` internal function verifies this role. Ensure the role management system is secure. Reentrancy is not directly applicable to mint/burn operations but ensure external token contracts do not introduce reentrancy risks. Input validation is performed by `checkTokenBridge` for the caller's role.
</Callout>
