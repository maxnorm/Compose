<DocSubtitle>
Handles burning of ERC721 tokens and updates enumeration.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Allows burning of ERC721 tokens.
- Maintains enumeration order by removing burned tokens from tracking.
</Callout>

## Overview

The ERC721EnumerableBurnFacet provides functionality to burn ERC721 tokens within a Compose diamond. It ensures that token destruction is correctly reflected in the enumeration tracking, maintaining the integrity of the token list.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC721EnumerableBurnFacet} from "@compose-protocol/diamond-contracts/contracts/facets/erc721/ERC721EnumerableBurnFacet.sol";

contract MyDiamond {
    IERC721EnumerableBurnFacet public erc721BurnFacet;

    function setErc721BurnFacet(address _facetAddress) external {
        erc721BurnFacet = IERC721EnumerableBurnFacet(_facetAddress);
    }

    function burnToken(uint256 _tokenId) external {
        // Access control for burning would typically be handled by a separate authorization facet
        // or within the diamond's router logic.
        erc721BurnFacet.burn(_tokenId);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `Transfer` event is emitted correctly upon successful burning, adhering to the ERC721 standard.
- Integrate this facet with an authorization mechanism to control who can burn tokens.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
Implement robust access control to prevent unauthorized token burning. Ensure that the `burn` function correctly handles all edge cases, such as attempting to burn a non-existent token (via `ERC721NonexistentToken` error) or tokens with insufficient approval (via `ERC721InsufficientApproval` error).
</Callout>
