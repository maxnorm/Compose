<DocSubtitle>
Burn ERC721 tokens within a Compose diamond.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Allows for the permanent destruction of ERC721 tokens.
- Integrates seamlessly with the Compose diamond pattern.
- Emits standard ERC721 `Transfer` event for burned tokens (from owner to address(0)).
</Callout>

## Overview

The ERC721BurnFacet enables the destruction of ERC721 tokens managed by a Compose diamond. It provides the `burn` function to remove tokens from existence and updates internal tracking to reflect their destroyed state. This facet is crucial for implementing token lifecycle management within an ERC721 compliant system.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IDiamondCut} from "@compose-protocol/diamond-contracts/contracts/interfaces/IDiamondCut.sol";
import {IERC721BurnFacet} from "./interfaces/IERC721BurnFacet.sol";

contract Deployer {
    address public diamondAddress;

    function deploy() public {
        // ... deployment logic for diamond proxy and facets ...
        diamondAddress = address(0xYourDiamondProxyAddress); // Replace with actual address
    }

    function burnToken(uint256 tokenId) public {
        IERC721BurnFacet(diamondAddress).burn(tokenId);
    }

    // Function to get the storage pointer (example)
    function getErc721Storage() public view returns (address storage) {
        return IERC721BurnFacet(diamondAddress).getStorage();
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the `ERC721BurnFacet` is properly initialized and its functions are accessible via the diamond proxy.
- Verify that the `burn` function is called with valid token IDs to prevent errors.
- Understand that burning a token is an irreversible action that permanently removes it from circulation.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The `burn` function should be protected by appropriate access control mechanisms (e.g., ERC721 owner or approved spender) to prevent unauthorized token destruction. Input validation on `tokenId` is critical to avoid the `ERC721NonexistentToken` error. Direct access to storage via `getStorage` should be done with extreme caution and only by trusted components.
</Callout>
