<DocSubtitle>
Manages diamond upgrades and facet installations.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Supports adding, replacing, and removing facets.
- Enables atomic initialization of new facets during upgrades.
- Provides functions to inspect diamond storage and owner.
- Enforces owner-only access for critical cut operations.
</Callout>

## Overview

The DiamondCutFacet provides the essential functionality for upgrading and modifying a diamond proxy. It allows for the addition, replacement, and removal of facets, enabling dynamic updates to the diamond's capabilities. This facet is crucial for the composability and evolvability of Compose diamonds.

## Usage Example

{`pragma solidity ^0.8.30;

import {IDiamondCut} from "@compose/diamond-contracts/contracts/facets/DiamondCutFacet.sol";

contract Deployer {
    address immutable diamondProxy;

    constructor(address _diamondProxy) {
        diamondProxy = _diamondProxy;
    }

    function upgradeDiamond(bytes[] memory facetCuts, address init, bytes memory initCalldata) external {
        IDiamondCut(diamondProxy).diamondCut(facetCuts, init, initCalldata);

## Best Practices

<Callout type="tip" title="Best Practice">
- Ensure the caller has the necessary ownership or permissions to execute `diamondCut`.
- Carefully validate facet addresses and selector ranges before performing cuts to prevent unintended changes.
- Use the `init` and `initCalldata` parameters for atomic initialization of new facets during an upgrade.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The `diamondCut` function is highly sensitive and should only be callable by authorized addresses (typically the diamond owner). Incorrectly specifying facet addresses or selectors can lead to denial of service or loss of functionality. Reentrancy is not a direct concern within `diamondCut` itself, but any `init` function executed via `delegatecall` must be carefully audited for reentrancy vulnerabilities.
</Callout>
