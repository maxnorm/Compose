Internal functions and storage for diamond proxy.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Manages facet registration and function selector mapping.
- Provides a centralized fallback for external function calls.
- Exposes diamond storage for introspection.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

The DiamondMod provides essential internal logic for managing diamond proxy facets and their associated storage. It handles adding new facets during deployment and provides a fallback mechanism to route function calls to the correct facet, ensuring composability and upgradability.

## Usage Example

## Best Practices

<Callout type="tip" title="Best Practice">
- Use `diamondFallback` for routing calls to facets; do not implement custom dispatch logic.
- Leverage `addFacets` only during initial diamond deployment to prevent unexpected state changes.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
DiamondMod interacts directly with the diamond's storage contract. The `addFacets` function is intended for use during the initial diamond deployment phase only. `diamondFallback` is the core routing mechanism; all external calls not directly implemented in the diamond proxy's facets will be routed through it. `getStorage` provides access to the raw storage data of the diamond.
</Callout>
