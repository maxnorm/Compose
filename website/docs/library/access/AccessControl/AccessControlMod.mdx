<DocSubtitle>
Manage role-based access control within a diamond.
</DocSubtitle>

<Callout type="info" title="Key Features">
- Enables granular permission management through roles.
- Supports granting, revoking, and checking role assignments for accounts.
- Allows dynamic configuration of role administration.
</Callout>

<Callout type="info" title="Module Usage">
This module provides internal functions for use in your custom facets. Import it to access shared logic and storage.
</Callout>

## Overview

This module provides a robust role-based access control (RBAC) system, enabling granular permission management for diamond functions. It ensures that only authorized accounts can perform sensitive operations, enhancing security and maintainability by decoupling permissions from specific facet implementations.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IAccessControlMod} from "@compose/modules/AccessControlMod.sol";

contract MyFacet {
    IAccessControlMod internal accessControl;

    // Assume accessControl is initialized elsewhere (e.g., in a diamond initializer)
    function setAccessControl(address _accessControl) external {
        accessControl = IAccessControlMod(_accessControl);
    }

    /**
     * @notice Grants the MANAGER role to an account.
     * @param _account The account to grant the role to.
     */
    function grantManagerRole(address _account) external {
        // Ensure the caller has the role that can grant MANAGER role (typically ADMIN)
        accessControl.requireRole(accessControl.getRoleAdmin(AccessControlMod.MANAGER_ROLE), msg.sender);
        accessControl.grantRole(AccessControlMod.MANAGER_ROLE, _account);
    }

    /**
     * @notice Checks if an account has the OPERATOR role.
     * @param _account The account to check.
     * @return bool True if the account has the role, false otherwise.
     */
    function doesAccountHaveOperatorRole(address _account) external view returns (bool) {
        return accessControl.hasRole(AccessControlMod.OPERATOR_ROLE, _account);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Store the `AccessControlMod` facet address securely and use it consistently across facets requiring authorization checks.
- Define custom roles with specific administrative roles using `setRoleAdmin` to create a clear hierarchy.
- Always use `requireRole` within facets to enforce access control before executing sensitive operations, reverting with `AccessControlUnauthorizedAccount` on failure.
</Callout>

## Integration Notes

<Callout type="success" title="Shared Storage">
The `AccessControlMod` stores role assignments and role admin mappings. Facets interact with this module via its interface. When `grantRole`, `revokeRole`, or `setRoleAdmin` are called, the state within the `AccessControlMod` facet is updated. Functions like `hasRole` and `requireRole` read from this state. It is crucial that the `AccessControlMod` facet is initialized and its address is correctly set for all interacting facets. Changes to role assignments or admin roles are immediately visible to all facets calling the respective read functions.
</Callout>
